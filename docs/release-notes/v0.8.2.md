# v0.8.2: fix Makefile install-dir detection

Changes since `v0.8.1`.

## Build / Install

### [Makefile](https://github.com/ssst0n3/fake-nvidia/blob/v0.8.2/Makefile) ([eef4205])

Fix:

- `make install` could place the `libnvidia-ml.so` stub in the wrong directory
  on toolchains where `gcc -print-multi-os-directory` returns a degenerate
  `../lib` (no architecture suffix), e.g. the Ubuntu 24.04-based
  `ghcr.io/ctrsploit/docker-v28.2.0:ctr_v0.1.0` image. The stub landed in
  `/usr/lib/` instead of the multiarch directory
  `/usr/lib/x86_64-linux-gnu/`, so the real driver library was not
  overwritten and `docker run --runtime=nvidia --gpus=all` failed with
  `failed to initialize NVML: Driver Not Loaded`. Affected since v0.7.3
  (commit `f029df3`); v0.7.2 and earlier hardcoded the multiarch path and
  were unaffected.

Root cause (two compounding defects in the Case-B branch):

1. `$(patsubst ../,%,$(...))` used a pattern without a `%` wildcard, so it
   matched only the whole word `../`, never a `../` prefix. For
   `../lib/x86_64-linux-gnu` the value was returned unchanged, giving
   `/usr/../lib/x86_64-linux-gnu`. The correct form is
   `$(patsubst ../%,%,$(...))`.
2. Even with the `patsubst` fixed, the compiler query is unreliable on the
   affected toolchain: `../lib` -> `lib` -> `/usr/lib`, still not the
   multiarch directory. So fixing only the `patsubst` did not resolve this
   case.

Resolution — the library directory is now detected in four layers, each
acting only if the previous yields nothing:

1. `LIBDIR` override (unchanged, highest priority).
2. **Dynamic linker resolution** of the already-installed real driver
   `libnvidia-ml.so.1` via `ldconfig -p`. The real driver library is
   installed before `make install` runs, so its location in the ldconfig
   cache is authoritative — the loader is the consumer the stub must
   shadow. Returns the multiarch directory on Debian/Ubuntu and
   `/usr/lib64` on RHEL.
3. Existence-ordered check of well-known multiarch directories
   (`/usr/lib/x86_64-linux-gnu`, `/usr/lib/aarch64-linux-gnu`,
   `/usr/lib64`); the aarch64 multiarch directory is now covered too.
4. `$(CC) -print-multi-os-directory` as a last resort, with the
   `patsubst ../%,%` fix applied.

A non-fatal post-install sanity check compares the directory
`libnvidia-ml.so.1` resolves to against `SHIM_INSTALL_DIR` and emits a loud
warning if they differ, so a future regression is obvious without failing
the install (the user may have intentionally used `LIBDIR`).
