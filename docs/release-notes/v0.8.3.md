# v0.8.3: fix Makefile install-dir detection for compat32 and cross-arch

Changes since `v0.8.2`.

## Build / Install

### [Makefile](https://github.com/ssst0n3/fake-nvidia/blob/v0.8.3/Makefile)

Fix:

- v0.8.2's loader-based detection could pick the **i386** `libnvidia-ml.so.1`
  entry and install the x86_64 stub into `/usr/lib/i386-linux-gnu/` when
  compat32 libs were present (e.g. after
  `NVIDIA-Linux-x86_64-*.run --install-compat32-libs`). The real x86_64
  driver library was left untouched and
  `docker run --runtime=nvidia --gpus=all` failed with
  `failed to initialize NVML: Driver Not Loaded`.

Root cause:

1. `$(lastword ...)` over the whole `ldconfig -p | grep` output is
   architecture-blind and order-dependent. With compat32 libs the i386
   entry sorts last, so `lastword` silently picked it.
2. The grep was not architecture-filtered: both the x86_64 and i386 SONAMEs
   match `libnvidia-ml.so.1`.

Resolution:

- The native architecture is now derived from the **compiler target triple**
  `$(CC) -dumpmachine` (`x86_64-linux-gnu` -> `x86-64`,
  `aarch64-linux-gnu` -> `aarch64`), not from `uname -m`. Under BuildKit
  cross-arch builds `uname` reflects the host kernel, not the image
  toolchain, and would select the wrong ldconfig entry.
- Layer 2 filters `ldconfig -p` to the native arch tag (e.g. `,x86-64`),
  ignoring compat32 (`libc6`) and x32 (`libc6,x32`) entries.
- The post-install sanity check now uses the **same** native-arch filter as
  the detection, so the two can no longer disagree on which entry is
  authoritative. A mismatch is **fatal** when `LIBDIR` was not set
  (auto-detection — a real regression) and **warn-only** when `LIBDIR` was
  set explicitly (the user asserted the directory, e.g. for `LD_PRELOAD`-
  only use).
