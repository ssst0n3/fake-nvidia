# v0.8.0: NVML symbols for nvidia-container-toolkit 1.19.0

Changes since `v0.7.4`.

## NVML Stub

### [fake_nvml.c](https://github.com/ssst0n3/fake-nvidia/blob/v0.8.0/fake_nvml.c) ([#17], [#18])

New:

- Exports `nvmlDeviceGetHandleByUUID` and `nvmlDeviceGetHandleByUUIDV`, which
  `nvidia-container-toolkit` / `libnvidia-container` >= 1.19.0 reference as
  undefined symbols resolved from `libnvidia-ml.so` via cgo. The stub previously
  exported only 17 NVML symbols, so toolkit 1.19.0 aborted with
  `symbol lookup error: undefined symbol: nvmlDeviceGetHandleByUUID`, breaking
  `nvidia-ctk cdi generate` / `cdi list`, `nvidia-container-runtime-hook`, and
  `docker run --runtime=nvidia --gpus=all`. `nvidia-container-toolkit` 1.17.8
  was unaffected because it does not reference these symbols.
  - Both APIs reverse-map a UUID back to a fake GPU handle (the mirror of
    `nvmlDeviceGetUUID`'s `GPU-<i>-FAKE-UUID`), returning `NVML_ERROR_NOT_FOUND`
    when no fake GPU matches; `NULL` arguments return `NVML_ERROR_INVALID_ARGUMENT`.
  - `nvmlDeviceGetHandleByUUIDV` is the versioned variant (capital V, not `_v2`)
    absent from older public `nvml.h` headers but exported by the real driver. It
    receives the UUID as a `nvmlUUID_t` struct whose value union holds an ASCII
    string or raw bytes; adds the `nvmlUUID_t` / `nvmlUUIDValue_t` /
    `nvmlUUIDType_t` typedefs and copies `value.str` into a NUL-terminated buffer
    so a binary UUID is compared safely and cannot match the ASCII fake UUIDs.
- Exports `nvmlDeviceGetPciInfo_v2`, `nvmlDeviceGetPciInfo_v3`, and
  `nvmlDeviceGetPciInfoExt`, also referenced as undefined symbols by toolkit
  >= 1.19.0. Without them every `docker run --runtime=nvidia --gpus=all ...`
  printed `Couldn't load symbol: ... undefined symbol: nvmlDeviceGetPciInfo_v3`
  (non-fatal, but noisy on every GPU run). All three fill the fake GPU's PCI
  info, consistent with `nvmlDeviceGetPciInfo`. Per `nvml.h`:
  - `_v2` / `_v3` take `nvmlPciInfo_t*` (same struct as the base API; `_v3` is
    the current default, header-aliased to `nvmlDeviceGetPciInfo`).
  - `Ext` takes the distinct `nvmlPciInfoExt_t*` (versioned struct with PCI
    base/sub class codes); adds the `nvmlPciInfoExt_v1_t` typedef, leaves
    `baseClass` / `subClass` zero, and preserves the caller-set `version` field.
- Exports the four MIG symbols that `libnvidia-sandboxutils.so` (shipped with
  toolkit >= 1.19.0) resolves from `libnvidia-ml.so.1` via `dlopen` + `dlsym`.
  Without them `nvidia-ctk cdi generate` printed
  `Couldn't load symbol: ... undefined symbol: nvmlDeviceGetMigDeviceHandleByIndex`
  (and later `nvmlDeviceGetDeviceHandleFromMigDeviceHandle`) followed by
  `Failed to init nvsandboxutils: ERROR_LIBRARY_LOAD; ignoring` (non-fatal —
  sandboxutils init failure is ignored and CDI generation still succeeds — but
  it polluted the output). The fake Tesla T4 does not support MIG, so there are
  no MIG device handles and no GPU/compute instances; all four return
  `NVML_ERROR_NOT_FOUND`. Signatures per `nvml.h`:
  - `nvmlReturn_t nvmlDeviceGetMigDeviceHandleByIndex(nvmlDevice_t device, unsigned int index, nvmlDevice_t *migDevice)`
  - `nvmlReturn_t nvmlDeviceGetDeviceHandleFromMigDeviceHandle(nvmlDevice_t migDevice, nvmlDevice_t *device)`
  - `nvmlReturn_t nvmlDeviceGetGpuInstanceId(nvmlDevice_t device, unsigned int *id)`
  - `nvmlReturn_t nvmlDeviceGetComputeInstanceId(nvmlDevice_t device, unsigned int *id)`

## CI

### [.github/workflows/release.yml](https://github.com/ssst0n3/fake-nvidia/blob/v0.8.0/.github/workflows/release.yml) ([4caa30f])

New:

- Adds a tag-triggered Release workflow (`push: v*` plus manual
  `workflow_dispatch`) that checks out the tagged commit, requires
  `docs/release-notes/<tag>.md` to exist, derives the release title and body
  from that file, runs a userspace shim compile smoke-test, and publishes a
  GitHub Release via `softprops/action-gh-release@v2`. Tags containing `-`
  (e.g. `v0.8.0-beta.3`, `v0.7.3-rc.1`) publish as pre-release; plain versions
  publish as latest. Releases are source-only (no portable binaries), matching
  the existing release convention.
